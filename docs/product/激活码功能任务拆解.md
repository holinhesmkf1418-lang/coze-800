# 激活码功能任务拆解

> 日期：2026-06-07  
> 目标：在暂不接支付的情况下，通过激活码开通会员权益，用于小红书售卖、机构合作、内测发码。

---

## 一、功能定位

激活码用于开通用户权益，不在小程序内直接收款。

推荐首版权益：

```text
30 天冲刺会员
```

首版只做兑换与权益判断，不做在线支付。

---

## 二、合规边界

小程序内只出现：

```text
已有激活码？输入兑换
```

不要在小程序内出现：

```text
去小红书购买
加微信转账
私信付款
扫码付款
```

原因：激活码可以作为权益凭证，但在小程序内引导站外交易或绕开平台支付，存在审核与平台规则风险。

小红书售卖侧需要按小红书商家后台允许的类目、商品描述、交付方式执行。建议商品页明确：

- 这是虚拟权益/激活码
- 无物流
- 交付方式为订单后提供激活码
- 退款规则
- 使用范围与有效期

## 三、后端任务

负责人：后端

### P0：数据库表

新增三张表：

```text
activation_codes
- id
- code
- planType
- durationDays
- maxUses
- usedCount
- startsAt
- expiresAt
- status
- note
- createdAt
- updatedAt

activation_redemptions
- id
- codeId
- userId
- redeemedAt

user_memberships
- id
- userId
- planType
- startsAt
- expiresAt
- source
- sourceCodeId
- status
- createdAt
- updatedAt
```

关键规则：

- `code` 唯一。
- `usedCount < maxUses` 才能兑换。
- `expiresAt` 过期后不能兑换。
- 同一个用户不能重复兑换同一个码。
- 如果用户已有会员，新会员有效期顺延。

### P0：API

新增接口：

```text
POST /api/activation/redeem
GET  /api/membership/status
```

#### `POST /api/activation/redeem`

Request：

```json
{
  "code": "GK800-ABCD-2026"
}
```

Response：

```json
{
  "code": 0,
  "message": "激活成功",
  "data": {
    "planType": "SPRINT_30",
    "expiresAt": "2026-07-07T00:00:00.000Z"
  }
}
```

错误场景：

| 场景 | 返回 |
|------|------|
| 激活码不存在 | `激活码无效` |
| 已过期 | `激活码已过期` |
| 已用完 | `激活码已被使用` |
| 用户已兑换过 | `你已兑换过该激活码` |

#### `GET /api/membership/status`

Response：

```json
{
  "code": 0,
  "data": {
    "isMember": true,
    "planType": "SPRINT_30",
    "expiresAt": "2026-07-07T00:00:00.000Z",
    "remainingDays": 30
  }
}
```

### P1：管理脚本

先不做后台页面，先做 CLI 脚本生成激活码：

```bash
npm run activation:generate -- --prefix GK800 --count 100 --days 30 --maxUses 1
```

输出 CSV：

```text
code,planType,durationDays,maxUses,expiresAt
GK800-7F2K-9Q3M,SPRINT_30,30,1,2026-12-31
```

## 四、前端任务

负责人：前端

### P0：兑换入口

入口放在：

```text
我的 → 设置 → 输入激活码
```

展示逻辑：

- 设置列表新增「激活码兑换」
- 点击后弹出输入框
- 输入后调用 `POST /api/activation/redeem`
- 成功后刷新会员状态

### P0：会员状态展示

在「我的」页头像区域或学习概览附近展示：

```text
冲刺会员 · 剩余 30 天
```

非会员展示：

```text
免费版
```

### P1：权益占位

先不强拦功能，只做展示：

- 首页展示会员状态
- 我的页面展示到期时间
- 后续再决定哪些功能限制免费版

## 五、PM / 测试任务

负责人：PM

### 测试用例

| 用例 | 预期 |
|------|------|
| 输入有效激活码 | 开通会员，显示到期时间 |
| 输入不存在的码 | 提示激活码无效 |
| 输入过期码 | 提示已过期 |
| 输入已用完码 | 提示已被使用 |
| 同一用户重复兑换 | 提示已兑换过 |
| 已有会员再兑换 | 有效期顺延 |
| 退出重进小程序 | 会员状态仍然存在 |

### 验收标准

```text
用户拿到激活码 → 在我的页输入 → 后端校验 → 开通 30 天会员 → 我的页显示会员状态
```

---

## 六、建议执行顺序

1. 后端先做数据库表、API、standalone 内存版接口。
2. 前端接「我的 → 激活码兑换」入口。
3. PM 用测试码完成兑换验收。
4. 后端补生成激活码 CSV 脚本。
5. 用户可在小红书售卖激活码，订单后手动发码。

---

## 七、首版不做

以下不进入首版：

- 小程序内支付
- 自动发码
- 商家后台
- 复杂优惠券
- 分销返佣
- 多等级会员

首版目标是验证：

```text
用户是否愿意为 30 天冲刺会员付费
```
